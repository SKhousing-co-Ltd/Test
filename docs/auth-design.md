# 基幹システム向け認証・権限設計

## 1. 認証方式

本システムの認証方式には **Supabase Auth** を採用する。

Supabase Auth により、メールアドレス・パスワード認証を基本とし、将来的な要件に応じて OAuth プロバイダーや多要素認証（MFA）の追加を検討できる構成とする。アプリケーション側では Supabase が発行するセッションおよび JWT を利用し、ログイン状態とユーザー識別を行う。

## 2. 想定ロール

本システムでは、以下のロールを想定する。

| ロール | 概要 |
| --- | --- |
| `admin` | システム全体の管理者。ユーザー管理、権限管理、全機能の操作を行える。 |
| `manager` | 業務管理者。担当範囲の業務データ確認、承認、マスタ情報の一部管理を行える。 |
| `staff` | 一般担当者。日常業務に必要なデータ登録・更新・参照を行える。 |
| `viewer` | 参照専用ユーザー。許可された情報の閲覧のみ行える。 |

## 3. ロールごとのアクセス可能機能

| 機能 | `admin` | `manager` | `staff` | `viewer` |
| --- | --- | --- | --- | --- |
| ダッシュボード閲覧 | 可 | 可 | 可 | 可 |
| 業務データ閲覧 | 可 | 可 | 可 | 可 |
| 業務データ登録 | 可 | 可 | 可 | 不可 |
| 業務データ更新 | 可 | 可 | 可 | 不可 |
| 業務データ削除 | 可 | 不可 | 不可 | 不可 |
| 承認・差戻し | 可 | 可 | 不可 | 不可 |
| マスタデータ閲覧 | 可 | 可 | 可 | 可 |
| マスタデータ登録・更新 | 可 | 一部可 | 不可 | 不可 |
| ユーザー管理 | 可 | 不可 | 不可 | 不可 |
| ロール・権限管理 | 可 | 不可 | 不可 | 不可 |
| 監査ログ閲覧 | 可 | 可 | 不可 | 不可 |
| システム設定 | 可 | 不可 | 不可 | 不可 |

※ 「一部可」は、業務要件に応じて管理対象のマスタを限定する。

## 4. フロント側の保護ルート方針

フロントエンドでは、認証状態とロール情報に基づいて保護ルートを制御する。

- 未ログインユーザーが保護対象ページへアクセスした場合は、ログインページへリダイレクトする。
- ログイン済みユーザーであっても、必要なロールを持たないページへアクセスした場合は、権限不足ページまたはダッシュボードへ遷移させる。
- 画面表示だけでなく、ナビゲーションメニューや操作ボタンもロールに応じて出し分ける。
- フロント側の制御はユーザー体験向上のための補助的な制御とし、最終的なアクセス制御は Supabase 側のポリシーで担保する。
- セッション失効時は Supabase Auth のセッション状態を検知し、ログインページへ誘導する。

## 5. Supabase 側の Row Level Security 方針

Supabase 側では、業務データを保持する各テーブルで **Row Level Security（RLS）を有効化する** 前提とする。

RLS ポリシーでは、Supabase Auth のユーザー ID とロール情報を利用し、テーブル単位および行単位でアクセス可否を制御する。フロントエンドからのリクエストは改ざん可能であるため、データベース側で必ず以下を担保する。

- ロールごとに許可された操作（SELECT / INSERT / UPDATE / DELETE）のみ実行できること。
- ユーザーがアクセス可能な組織・部門・担当範囲のデータのみ参照・更新できること。
- `viewer` は参照系操作のみ許可し、更新系操作を許可しないこと。
- `admin` であっても監査要件に応じて操作ログを記録できること。

これにより、フロント側の保護ルートと Supabase 側の RLS を組み合わせ、多層的な認証・認可を実現する。

## 6. アカウントと担当者の紐づけ

- `auth.users` と `employee_master` は `user_profiles` を介して最大1対1で紐づける。
- 従業員マスタのメールアドレスと Auth のメールアドレスが一致し、在籍状態が `active` の場合、Auth 登録トリガーが `staff`・`active` のプロフィールを自動作成する。
- 一致しない登録は `pending` の仮アカウントとして作成し、`admin` が担当者・ロールを設定して有効化するまで業務データへアクセスさせない。
- `employee_master.employment_status` が `inactive` になった場合、紐づくアカウントは `suspended` に同期する。過去の業務履歴は削除しない。
- 初回の管理者は、対象アカウント作成後に Supabase SQL Editor で `user_profiles.role` を `admin` に更新して設定する。
